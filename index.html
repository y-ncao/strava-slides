Why Agent Security Became A Platform Problem

What We Did: Built a reusable foundation for production AI agents: agents get identities, users delegate work, tool access is authorized, credentials managed via Vault, and the User -> Agent -> Resource chain is auditable.

Why Now

  • Agents autonomously calling MCP, internal APIs, and 3rd-party resources.
  • Agent adoption reached production scale across multiple runtimes.
  • Legacy IAM (users/services) not built for delegation or dynamic tools.

My Role (Senior EM)

  • Accountable for platform architecture direction; 30-person XFN effort.
  • Defined trust boundaries, ownership, and aligned 15+ XFN partners.
  • Owned Identity across human, service, and agent domains, plus Token Vault.

Our Solutions

Identity Delegation Confusion -> First-class Agent Identity

Give agents independent first-class identities and natively support a trusted Human -> Agent delegation chain on top of unified identity.

Permission Overreach -> Dynamic & Fine-Grained AuthZ

Context-aware permission model reaches the tool / data-domain level.

Credential Leakage -> Centralized Custody

Outbound tokens go through Token Vault; short-lived scoped credentials.

Poor Traceability -> Full-Chain Audit

User -> Agent -> Resource path is traceable for accountability and compliance.

How We Got It Done

Roadmap Direction

  • Internal AI Infra and Agent Developers
  • Public Cloud for Serve ToB customers

Ownership Map

  • Direct Report Team: Identity (Human, Service, Agent) & Token Vault.
  • Peer Engineering Team: AuthZ & Gateway.
  • Close XFN: SRE / QA / frontend / design / TPM / PM.
  • Extended XFN: ToB Sales, internal & external cloud infra, security partners, PR / Legal / Privacy.

Operating Model

  • Strategy and success criteria: Agent Identity as a default security foundation.
  • Ownership and accountability: clear owners across identity, AuthZ, vault, gateway, SRE/QA, PM, and TPM.
  • Architecture decision process: irreversible contracts first: namespace, delegation, credentials, policy, and audit.
  • Execution rhythm: TPM tracking, EM/TL reviews, PM prioritization, and launch readiness.
  • Learning loop: H2 retro actions on cadence, ownership, dependencies, freeze, intake, and stability drills.

Key Tradeoff

Avoided a one-off feature for one platform. Goal was a reusable trust foundation.

Stumbling Block

Shifted to OpenClaw ecosystem integration (skills/hooks) without losing core runtime control.

Shipped OpenClaw path quickly, but kept identity, delegation & audit as reusable contracts.

Technical Architecture And Trust Boundary

  • Federated User Identity: Enterprise/3rd-party IdP federates into UserPool.
  • Agent Identity: First-class identities, no user/service account impersonation.
  • Delegation (TIP): Trusted Identity Propagation carries user + agent context.
  • Permission Management: Authorize by trust chain + resource/action/context.
  • Credential Mgmt: Centralized OAuth, API keys, cookies.
  • Inbound/Outbound Control: Gateway enforces rules directly on the traffic path.
Important distinction: Inbound = "Who can call this agent?" | Outbound = "What can this agent do?" - Outbound control is where most damage is stopped.

Results & Learnings

Results

  • Shared model for agent identity, delegation, permissions, vaulting, and audit.
  • Integrated MCP/tool ecosystems, OpenClaw-style agents & cloud surfaces.
  • Aligned 9+ functions around one trust foundation.
  • Enterprise POCs (50K+ employees) & production readiness (multi-AZ, rate limits).

Lessons Learned

  • Platform path is mandatory; SDK-only controls are too easy to bypass.
  • Identity semantics must be unified early before platforms invent their own.
  • Product adoption requires reducing developer burden, not just adding gates.
  • EM challenge: balance platform coherence with urgent customer asks.
